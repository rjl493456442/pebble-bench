# pebble-bench

A benchmarking tool for the [Pebble](https://github.com/cockroachdb/pebble) key-value
store. It populates a dataset and runs scan / read / write / mixed workloads,
reporting latency percentiles and Pebble's internal metrics (compactions, write
stalls, cache hit rates, etc.).

It can benchmark **both Pebble v1 and Pebble v2** from a single binary — pick the
backend at runtime with the `--pebblev2` flag.

## Build

```sh
go build -o pebble-bench .
```

## Quick start

```sh
# 1. Populate a 1GB dataset
./pebble-bench init --data-dir /tmp/pb --target-size 1GB

# 2. Run a random-read benchmark for 5 minutes with 4 workers
./pebble-bench run --data-dir /tmp/pb -b read -d 5m --concurrency 4

# 3. Run the same against Pebble v2
./pebble-bench init --pebblev2 --data-dir /tmp/pb-v2 --target-size 1GB
./pebble-bench run  --pebblev2 --data-dir /tmp/pb-v2 -b read -d 5m --concurrency 4
```

## Commands

### `init` — populate a dataset

Fills a Pebble database with deterministic key/value data up to a target size.
Keys are generated by index (sha256-based), so read benchmarks regenerate them
without storing anything in memory. Running `init` again on an existing dataset
extends it toward the new target.

```sh
./pebble-bench init --data-dir /tmp/pb --target-size 200GB --value-size 256
```

| Flag | Description |
|------|-------------|
| `--target-size` | Target dataset size, e.g. `1GB`, `200GB`, `512MB` |
| `--key-size` | Key size in bytes (default from config) |
| `--value-size` | Value size in bytes (default from config) |
| `--batch-size` | Write batch size during population (default `1000`) |

### `run` — execute a benchmark

```sh
./pebble-bench run --data-dir /tmp/pb -b mixed -d 2m --concurrency 8
```

| Flag | Description |
|------|-------------|
| `-b, --benchmark` | Benchmark name: `scan`, `read`, `write`, `mixed` |
| `-d, --duration` | Run duration, e.g. `30s`, `5m` |
| `--num-ops` | Stop after N operations (alternative to `--duration`) |
| `--concurrency` | Number of worker goroutines |
| `--output` | Output format for `--output-file`: `terminal` (default), `json`, `markdown` |
| `--output-file` | Write results to a file |

Available benchmarks:

- **scan** — iterate over keys with a Pebble iterator (measures `Next` throughput)
- **read** — random point reads by key
- **write** — batched key/value writes
- **mixed** — configurable read/write mix (see `read_percent`)

### `compare` — diff two result files

```sh
./pebble-bench run -b read -d 5m --output json --output-file v1.json
./pebble-bench run -b read -d 5m --pebblev2 --output json --output-file v2.json
./pebble-bench compare v1.json v2.json
./pebble-bench compare --output-file diff.md v1.json v2.json
```

## Pebble v1 vs v2

By default benchmarks run against Pebble **v1**. Pass `--pebblev2` (available on
both `init` and `run`) to use Pebble **v2** instead:

```sh
./pebble-bench run --pebblev2 --data-dir /tmp/pb-v2 -b write -d 1m
```

The same configuration is translated to each backend's options, so results are
directly comparable. Note that a dataset created with one version should be
benchmarked with that same version (use separate `--data-dir`s for v1 and v2).

## Configuration

All settings have sensible go-ethereum-inspired defaults. To customize, pass a
YAML config file or override individual keys on the command line.

### Shared flags (all commands)

| Flag | Description |
|------|-------------|
| `-c, --config` | Path to a YAML config file |
| `--override key=value` | Override a single config key (repeatable) |
| `--data-dir` | Override the data directory |
| `--log-file` | Also write log output to this file |
| `--pebblev2` | Use the Pebble v2 backend instead of v1 |

### Config file

See [`config.example.yaml`](config.example.yaml) for the full set of options
(cache size, memtable, compaction, sync, bloom filter, and per-level settings).

```sh
./pebble-bench run -c config.yaml -b read
```

### Inline overrides

```sh
./pebble-bench run \
  --override cache_mb=4096 \
  --override no_sync=false \
  --override benchmark.read_percent=90 \
  -b mixed -d 1m
```

Supported override keys include: `data_dir`, `cache_mb`, `max_open_files`,
`pebble_v2`, `mem_table_size`, `mem_table_count`, `max_concurrent_compactions`,
`l0_compaction_threshold`, `bytes_per_sync`, `wal_bytes_per_sync`, `disable_wal`,
`no_sync`, `bloom_filter_bits`, and `benchmark.*` (`name`, `duration`,
`concurrency`, `num_ops`, `key_size`, `value_size`, `batch_size`, `read_percent`,
`init_target_size`). Run any command with an invalid key to see the full list.

## Environment variables

Optional diagnostic logging, useful when investigating compaction or write-stall
behavior:

| Variable | Effect |
|----------|--------|
| `PEBBLE_BENCH_SLOW_FLUSH_THRESHOLD` | Log flushes slower than this duration (e.g. `5s`) |
| `PEBBLE_BENCH_LOG_COMPACTION` | Set to `1`/`true` to log compaction begin/end |
| `PEBBLE_BENCH_LOG_WRITE_STALL` | Set to `1`/`true` to log write-stall begin/end |

## Output

`run` always prints a summary to the terminal: throughput (ops/sec), latency
percentiles (p50/p95/p99/p99.9), and a final snapshot of Pebble metrics. During
the run it logs periodic ticks (every 3s) with interval throughput and live
Pebble stats. Use `--output-file` with `--output json|markdown` to persist
results for later comparison.

### Key performance indicators

The summary reports the metrics most useful for comparing engines/versions
(modeled on Pebble's own `bench ycsb` output):

- **Ops/sec** — overall throughput.
- **Latency percentiles** — especially **p99 / p99.9 / max** (tail latency,
  where write stalls and compaction interference show up). For the `mixed`
  benchmark, latency is also broken down **per operation** (read vs write), since
  their tails differ greatly.
- **Write Amp** — write amplification = bytes written to disk ÷ logical bytes
  written. The single most telling metric for LSM compaction efficiency. Shown
  alongside the raw **Bytes Written / Read / logical-in**.
- **Read Amp** — reported as `final (avg, max)`, averaged over the whole run
  rather than just the ending snapshot.
- **Sync calls (VFS)** — counts and avg/max timing of the durability syscalls
  Pebble issues, broken out as **fsync**, **fdatasync**, and **sync_file_range**.
  These are measured in-process by instrumenting Pebble's VFS layer (no strace or
  eBPF needed), so they work the same on Linux and macOS. Note `fdatasync` and
  `sync_file_range` are Linux-only; on macOS those counts are 0 because Pebble
  falls back to `fsync` there.
- **Read calls (VFS)** — counts and avg/max timing of read syscalls, broken out
  as **pread** (`ReadAt`, the dominant path for sstable block reads) and **read**
  (`Read`, sequential reads). Measured at the same VFS layer, so the counts only
  include reads that **miss the block cache and reach the disk** — making `pread`
  avg a good proxy for the average disk-read latency, especially for the `read`
  and `scan` benchmarks. (If the dataset fits in the OS page cache, these will be
  fast page-cache hits rather than device reads.)

`compare` diffs all of these side by side with percentage deltas.
